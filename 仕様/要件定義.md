# 1. プロジェクト概要：Minecraft Spatial Audio Link

**プロジェクト目的:**
Minecraft (Paper) サーバーのプレイヤー位置、視線、および活動状態を、リアルタイムで外部 Web サイトの音声処理に反映させ、没入感の高いボイスチャット環境を提供する。

---

# 2. システムアーキテクチャ

システムは 4 つの主要コンポーネントで構成される。



1.  **Minecraft 側 (Paper Plugin):** 座標情報の送信機。
2.  **データ基盤 (Redis):** 超高速な位置情報のキャッシュ。
3.  **バックエンド & 管理 (Django):** 認証、権限管理、ユーザーインターフェース。
4.  **音声基盤 (LiveKit SFU):** 音声パケットの低遅延転送。
5.  **クライアント (Web Audio API):** 各自のブラウザによる音声のミキシング・エフェクト処理。

---

# 3. 機能要件 (Functional Requirements)

### 3.1 近接チャット (Proximity Chat)
* **有効範囲:** 半径 50m。
* **音響特性:** 3D立体音響（HRTF）。距離に応じた減衰および高域カット（Low-pass Filter）。
* **座標更新:** 200ms〜500ms 間隔で更新。

### 3.2 管理者用全体放送 (Global Broadcast)
* **権限者:** サーバー管理者（LuckPerms 権限保持者）および Sawamura（専用 WebUI）。
* **特性:** 距離減衰なし、ノイズなし、ステレオ中央配置。
* **優先順位:** 放送中は近接チャットの音量を自動で微減（ダッキング機能）。

### 3.3 ラジオ機能 (Radio / Walkie-Talkie)
* **有効範囲:** 最大 2km (2000m)。
* **チャンネル:** 1〜999 までの整数チャンネル。
* **距離ノイズ:** 0.5km を超えたあたりからホワイトノイズが混入し、2km 付近で S/N 比が最小となる。
* **フィルタ:** ラジオらしい質感（High-pass Filter）を適用。

# 4. 技術仕様 (Technical Specifications)

### 4.1 音声ミキシング・ロジック (Client-side)
サーバー負荷軽減のため、ミキシングはすべてブラウザ側で行う。



#### 4.1.1 距離計算式
3次元空間における 2 点間の距離 $d$ を算出する：
$$d = \sqrt{(x_2 - x_1)^2 + (y_2 - y_1)^2 + (z_2 - z_1)^2}$$

#### 4.1.2 ラジオ用ノイズ計算
ラジオにおけるノイズ利得 $G_{noise}$ は、2km を最大距離 $D_{max}$ としたとき、以下のモデルを適用する：
$$G_{noise} = \begin{cases} 0 & (d < 500) \\ \frac{d - 500}{D_{max} - 500} \times K & (500 \le d \le 2000) \\ K & (d > 2000) \end{cases}$$
※ $K$ は最大ノイズ定数。

---

### 4.2 データ構造 (Data Design)

#### 4.2.1 Redis (Real-time Store)
座標データは永続化せず、TTL（有効期限）を設定したキャッシュとして保持。
* **Key:** `player_pos:{uuid}`
* **Value (JSON):**
    ```json
    {
      "pos": [x, y, z],
      "yaw": 0.0,
      "world": "world",
      "channel": 105,
      "mode": "spatial|global|radio"
    }
    ```

#### 4.2.2 Django (Database Models)
ユーザー管理と認証トークンを管理。
* **UserLink:** Minecraft UUID, Discord ID (Optional), Django User ID.
* **BroadcastPrivilege:** 管理権限を持つプレイヤーのリスト。
* **SessionToken:** ゲーム内 `/vc join` 時に発行される使い捨てトークン。

# 5. インフラストラクチャ詳細

* **環境:** Proxmox 上の仮想マシン（Ubuntu Server 推奨）。
* **ネットワーク:** * **二重ルーター対策:** Cloudflare Tunnel (`cloudflared`) を使用し、外部からの 443 ポート（HTTPS/WSS）を Django へルーティング。
    * **SFU 通信:** LiveKit は UDP ポートを使用するため、必要に応じてルーターのポートマッピング（DNAT）を 2 段階で設定。
* **ドメイン:** `clusters-prj.com` のサブドメイン（例: `vc.clusters-prj.com`）を割り当て。

# 6. 制御フロー (Sequence Diagram)

### 6.1 認証フロー
1.  プレイヤーがゲーム内で `/vc join` を実行。
2.  Plugin が Django API にリクエストし、6 桁のトークンを生成。
3.  プレイヤーがブラウザで Django サイトにアクセスし、トークンを入力。
4.  Django がセッションを確立し、LiveKit の接続 JWT (Token) を発行。

### 6.2 放送制御フロー (Sawamura 専用 WebUI)
1.  Sawamura が WebUI の「Broadcast START」ボタンを押下。
2.  Django が WebSocket 経由で全クライアントに `audio_mode: global` をブロードキャスト。
3.  全クライアントが Sawamura の Audio Stream を `PannerNode` からバイパス（切替）。

# 7. 開発ロードマップ (Roadmap)

### フェーズ 1: 骨格形成
* Paper プラグインから Redis への座標書き込み実装。
* Django と Redis の連携。
* Cloudflare Tunnel による外部アクセス環境の構築。

### フェーズ 2: 音声基盤構築
* LiveKit サーバーのデプロイ。
* ブラウザ間での音声送受信テスト（まだ減衰なし）。

### フェーズ 3: 音響処理実装
* Web Audio API による距離減衰の実装。
* ラジオ用 2km 減衰・ノイズモデルの実装。

### フェーズ 4: 拡張機能
* 全体放送コマンドの実装。
* Sawamura 専用管理パネルの構築。
* ラジオノイズのエフェクト微調整（ロジャービープ音など）。
