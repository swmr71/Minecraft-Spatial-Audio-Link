# Minecraft Spatial Audio Link：要件定義書（最終案）

## 1. プロジェクト概要
Minecraft（Paper）のゲーム内状況を WebRTC 技術を用いてブラウザ音声に完全同期させ、究極の没入感と実用性（広域無線・全体放送）を両立したボイスチャットシステムを構築する。

## 2. システム構成・役割分担

| コンポーネント | 役割 |
| :--- | :--- |
| **Paper Plugin** | 座標 $(x, y, z)$、視線 ($Yaw$)、世界 ($World$)、使用チャンネルの取得と Redis への送信。 |
| **Redis (In-memory)** | 数百ミリ秒周期で更新される全プレイヤー状態の超高速キャッシュ基盤。 |
| **Django (Backend)** | ユーザー認証、管理者権限管理、専用 WebUI の提供、SFU 接続トークン発行。 |
| **LiveKit (SFU)** | 音声パケットの中継。サーバー側 VAD による無音パケットの破棄。 |
| **Web Browser** | クライアント側 VAD、空間音響、距離減衰、ノイズ合成、ワールド越境処理。 |

---

## 3. 音声通信モード仕様

### 3.1 近接チャット (Proximity)
* **範囲:** 半径 50m 以内。
* **音響:** 3D 立体音響（HRTF 採用）。
* **減衰:** 距離 $d$ に応じて音量を下げ、高域をカット（空気吸収の再現）。
* **購読条件:** 同一ワールド かつ 距離 $d \le 100m$（余裕を持たせたバッファ）。

### 3.2 管理者全体放送 (Global Broadcast)
* **トリガー:** ゲーム内 `/vc broadcast` または Sawamura 専用 WebUI ボタン。
* **音響:** 距離・方向に関わらずステレオ中央で再生。ノイズ・減衰なし。
* **割り込み:** 放送中は他モードの音量を 30% 自動ダウン（ダッキング）。

### 3.3 ラジオ機能 (Radio / World-Crossing)
* **範囲:** 最大 2km (2000m)。
* **越境:** ワールドが異なる場合、距離 $d$ を一律 2000m として処理。
* **ノイズ:** 500m を超えると S/N 比が悪化。2km 付近では 90% がノイズになる。
* **質感:** バンドパスフィルタによる無線機特有の音質加工。

## 4. 最適化・パフォーマンス仕様（二重VAD）

サーバーおよびクライアントの負荷を最小化するため、以下の重層的な最適化を行う。

### 4.1 二重 VAD (Voice Activity Detection)
1.  **第一段階（クライアント側）:** ブラウザがマイク入力を常に監視。一定の入力レベル（閾値）を下回る場合、SFU への音声パケット送信を物理的に停止する。
2.  **第二段階（サーバー側）:** SFU (LiveKit) が各ストリームの信号を分析。無音と判断されたパケットは、購読している他ユーザーへ転送せず破棄する。

### 4.2 音声購読の動的最適化 (Selective Subscription)
* 全プレイヤーの音声を受信するのではなく、Django から通知された「今聞くべきリスト（近くの人 ＋ 放送者 ＋ ラジオ仲間）」の UUID だけを SFU から購読する。

## 5. 音響物理モデル（詳細計算）

### 5.1 距離計算
同一ワールド内におけるプレイヤー間の距離 $d$：
$$d = \sqrt{(x_2 - x_1)^2 + (y_2 - y_1)^2 + (z_2 - z_1)^2}$$

### 5.2 ラジオノイズ利得 ($G_{noise}$)
距離 $d$ に基づくノイズ混入率（$D_{max} = 2000$）：
$$G_{noise} = \begin{cases} 0 & (d < 500) \\ \frac{d - 500}{D_{max} - 500} & (500 \le d \le 2000) \\ 1.0 & (d > 2000) \end{cases}$$

## 6. セキュリティ・管理仕様
* **認証:** ゲーム内で生成された 1 回限りのトークンを用いた Web ログイン。
* **専用 UI:** `clusters-prj.com` 傘下の特定 URL に Sawamura 専用の管理パネル（全チャンネル傍受・強制放送ボタン）を設置。
* **インフラ:** * Proxmox 上の VM に展開。
    * 二重ルーター対策として **Cloudflare Tunnel** を使用。
    * 内部通信は Redis Pub/Sub で高速連携。

---

## 7. 開発・運用における制約
* 画像生成ツール（Nano Banana 等）は使用せず、技術ドキュメントとコードベースで進行する。
* サーバー本体 (10.2.0.x) および仮想物 (10.2.x.y) の IP 体系を遵守して構成する。
