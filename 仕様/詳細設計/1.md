## 1. Redis データ構造定義（リアルタイム座標）

座標データは1秒間に数回更新されるため、Redisの `SET` コマンドで上書き保存します。キー名はUUIDを用いて一意に特定します。

### キー形式
`vchat:player:{UUID}`

### JSONスキーマ
```json
{
  "u": "String (UUID)",       // プレイヤーのUUID
  "n": "String",              // プレイヤー名
  "p": [x, y, z],             // 座標（小数点以下2桁まで）
  "y": Float,                 // Yaw (0.0 - 360.0)
  "w": "String",              // ワールド名 (world, world_nether, etc.)
  "c": Integer,               // ラジオチャンネル (0 = OFF)
  "m": "String",              // 現在のモード (spatial | global | radio)
  "t": Long                   // 送信時刻（サーバー側での遅延判定用）
}
```
> **最適化メモ:** 通信量を削るため、キー名を1文字（u, p, yなど）に短縮しています。

## 2. Plugin ➔ Redis 送信仕様

Paperプラグイン側での実装方針を固めます。

* **送信トリガー:** `Scheduler` を用いて **5Ticks（250ms）毎**に実行。
    * *理由:* 20Ticks（毎秒）だとブラウザ側での補間が厳しく、1Tick（50ms）だとRedisへの書き込み負荷が高すぎるため。
* **非同期処理:** メインスレッドを止めないよう、`BukkitRunnable#runTaskAsynchronously` でRedisへ書き込む。
* **生存確認 (TTL):** Redisのキーに **5秒の有効期限** を設定。プラグインやサーバーがクラッシュした際、古い座標で声が聞こえ続けるのを防ぎます。

## 3. Django ➔ ブラウザへの通知（WebSocket / Socket.io）

ブラウザが「誰の座標を追うべきか」を判断するためのフローです。

1.  **購読リストの作成:**
    Django Channels が各クライアントに対し、**「半径200m以内のプレイヤーUUIDリスト」**および**「同一ラジオチャンネルのプレイヤーUUIDリスト」**を定期的に配信します。
2.  **差分更新:**
    リストに変化（新規入室・退出）があった場合のみ通知することで、WebSocketのトラフィックを抑えます。

## 4. 認証APIの詳細（プラグイン ↔ Django）

プレイヤーがWebサイトへ安全にログインするための仕組みです。

| エンドポイント | メソッド | 説明 |
| :--- | :--- | :--- |
| `/api/vc/token/generate/` | POST | プラグインがUUIDを送り、6桁の数字トークンを取得。 |
| `/api/vc/token/verify/` | POST | Web側でユーザーが入力したトークンを照合し、Sessionを発行。 |
